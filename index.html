Hoplon

A simpler way to program the web.

Get started »

Clojure, everywhere.

Write everything in Clojure and ClojureScript, clientside and serverside. Even the page markup is ClojureScript that is evaluated to produce the DOM. Pages can still be written using the familiar HTML syntax if desired.

Simplicity on the client.

Use a spreadsheet-like dataflow programming environment to manage client state. Develop basic content-oriented pages and complex thick-client applications without descending into “callback hell”.

Simplicity on the server.

Decouple business logic from HTTP-related concerns while satisfying complex, fine-grained authorization requirements with transparent, performant transfer of Clojure data between server and client.

Pages The DOM as a program.

Imagine a programming environment where variables cannot have names. Instead they have “properties” and are stored in a global tree structure. Then, whenever the program wants to access a variable it must use “selectors” to walk this tree, find it by matching property values, and obtain a reference that can be used to manipulate it.

Insane, you say? Unfortunately, this is the state of affairs that confronts frontend developers every day.

A serious obstacle to achieving simplicity in frontend development is the disconnect between the DOM and the program that runs in the client to manipulate it. The DOM is a completely separate entity that must be examined from outside, using CSS selectors to find specific elements. Worse yet is the syntactic and semantic disconnect between page markup and the DOM programming interface. A program can combine DOM elements to form “widgets” and give them names by which the program can refer to them, but there is no literal representation for them in markup, for instance.

Hoplon unifies the two, providing an environment where page markup and DOM programming are interchangeable, syntactically and semantically. In a Hoplon application the page document is the result of evaluating a ClojureScript expression, not something that is inserted into the page separately and then manipulated from outside. The elements in the DOM are organically created by the program, making it unnecessary to use selectors to find them later, just like a regular computer program.

Example:

(page "index.html")

(defn list-widget [title & items]
  (div
    :class "list-widget"
    (title)
    (apply
      (ul :class "list-widget")
      (map #(li :class "list-widget-item" %) items))))

(html
  (head)
  (body
    (list-widget
      (p "Item 1.")
      (p "Item 2."))))

Spreadsheets Client state made simple.

Remember how awesome spreadsheets are? The most accessible programming environment ever devised, spreadsheets are great because they provide a totally intuitive model for application state, leaving the developer free to think in terms of values, formulas, and the relationships between the two. Then, when the underlying cells are set up, forms and charts are added and linked to the cells to create user interfaces that manage their own state, automatically.

Hoplon applies this crazy, futuristic technology to frontend web development. Forgoing the clunky grid metaphor and names like A1 and B27, Hoplon provides a cell reference type. These cells, like cells in a spreadsheet, are either updated directly (input cells), or automatically in accordance with a formula.

Example:

(defc a {:a 20})              ;; define input cell
(defc b {:b 20})              ;; define input cell
(defc= c (+ (:a a) (:b b) 2)) ;; define formula cell
@c                            ;;=> 42
(swap! a update-in [:a] inc)  ;; increment value in input cell
@c                            ;;=> 43
